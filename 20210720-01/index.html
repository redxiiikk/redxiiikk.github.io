有限状态机

Posted on 2021-07-21 Edited on 2021-10-20 In 算法 Disqus:

什么是有限状态机？

维基百科上关于有限状态机的定义如下：

有限状态机是表示有限个状态及其和状态之间的转移和动作等行为的数学模型。

​

为什么要使用有限状态机？

在思考这个为题之前我们先来想一想如果不使用有限状态机是如何进行状态之间的转移的？最简单的方式就是 if else 进行判断，如果当前的状态满足了某一个 if 条件，就执行其中定义好的代码。

这样的做法简单易懂，但是随着时间的推移和业务的不断变化，就会导致 if else 数量不断的增多，所带来的的影响就是 BUG 的频繁出现和维护成本的增加；同时 if else 的大量使用会导致陷入逻辑泥潭之中，无法有效的在代码中沉淀业务。

那么针对于上述的描述，有限状态机的好处显而易见：

  1. 增加系统架构的可扩展性；
  2. 降低后期的维护成本；
  3. 可以将业务知识沉淀在代码中（业务知识极其的宝贵）。

有限状态机的组成

个人对于有限状态机所做的事情总结如下：

客户端发出的 事件A 导致 状态B 在满足 条件C 的前提下执行 动作D 后转化到 状态E。

从上述的描述话语中我们可以了解到有限状态机有四部分组成元素，分别是事件、状态、条件以及动作。

事件

事件是状态转移的发起者。触发事件的客户端存在多种类型，可能是一个真实的用户，也可能是某一定时任务。

事件通常由两部分组成：第一部分是事件类型，一般与一种状态转移相关；第二部分是参数，是执行状态转移及其转移动作的过程中所必须的数据，参数可以有客户端直接发送，也可以从系统中获取。

状态

实体对象处于某种上下文的一个标志，可以使用字符串或者枚举来进行表示。

条件

是指状态转移的前提，是必须要满足的业务意义上的要求。

动作

对于动作我们可行进一步的细分：

  1. 进入动作：特指在进入该状态时必须要发生的动作，与转移前状态无关；
  2. 转移动作：指在状态转移的过程中发生的动作，与转移前和转以后的状态相关；
  3. 离开动作：特指在离开该状态时必须发生的动作，与转以后的状态无关。

有限状态机的两种建模方式

建模方式是指我们以何种方式去表示有限状态机，目前主流的存在两种方式，分别是状态转移表和状态图。

状态转移表

状态转移表是指以表格的形式来表示有限状态机的方式，通常存在多种形式。

状态图

以有向图的方式来表示有限状态机，也是最为常用的表示有限状态机的方式。

在状态图中，节点表示状态，边表示转义动作，并且在边上我们还会标注出本次转移所关联的其他的元素（事件、条件和动作）。

设计模式中的状态模式

设计模式中的状态模式与有限状态机存在一定的相似性，接下来我们可以了解一些状态模式，然后对两者进行比较。

状态模式

状态模式允许对象在内部状态改变时修改它的行为，对象开起来好像修改了它的类。

有限状态机和状态模式之间的比较

  1. 有限状态机关注在状态转移的过程；
  2. 状态模式则关注在特定的状态下类自身的行为。

有限状态机与状态模式的结合

  1. 有限状态机负责状态转移；
  2. 状态模式负责负责不同状态下类的行为；
  3. 在进入动作中切换类的行为。
